# RPG_Ero — Движок создания персонажа (`engine/chargen`) — дизайн

**Дата:** 2026-06-30
**Статус:** согласован, готов к плану реализации
**Контекст:** ядро Вехи 1 (см. [2026-06-25-rpg-ero-design.md](2026-06-25-rpg-ero-design.md) §6). Датасет правил уже собран из pf2.ru (Remaster-only): `backend/data/{ancestries,classes,backgrounds}.json`.

## 1. Замысел и охват

Чистый движок создания персонажа PF2e Remaster **1-го уровня по книге**. Класс-нейтральное ядро: считает характеристики, владения, производные статы, валидирует выборы, экспортирует лист. Цель по контенту — **все 21 класс, включая заклинателей**, но это достигается инкрементально: движок потребляет данные через схему и работает для любого класса, у которого данные заполнены.

**Декомпозиция (потоки работы):**
1. **Ядро движка** (этот спек) — класс-нейтральное, чистое, тестируемое на фикстурах.
2. **Данные классов** — начальные владения + классовые особенности 1 ур. всех 21 класса (извлечение прозы детальных страниц) — отдельные планы.
3. **Заклинания** — списки/слоты/традиции/фокус — отдельные планы; в ядре заложена точка расширения.
4. **Хранилище правил (`rules`-схема + сидер)** — инфра-поток, идёт рядом с ядром (ядро на фикстурах его не ждёт).

**Вне охвата v1 движка:** уровни выше 1-го; снаряжение/золото и доспешный AC (отдельный поток); выбор заклинаний (заглушка-точка расширения); полное наполнение данных всех 21 класса (потоки 2–3).

## 2. Хранение данных

**Правила (канон): PostgreSQL, отдельная схема `rules`.** Таблицы `ancestries`, `heritages`, `backgrounds`, `classes`, `feats` (+ `spells` позже). SQLAlchemy-модели + Alembic-миграции. Пользовательские данные (`users`, `characters`, `game_states`) — отдельно (схема `public`/`app`), приходят с API/персистентностью (последующий поток).

**Сид-источник:** конвейер pf2.ru продолжает писать версионированный `backend/data/*.json` (воспроизводимый git-канон, оффлайн-тесты). **Идемпотентный сидер** заливает JSON → схему `rules` (на деплое/командой). Так правила в БД в рантайме, но снимок-канон остаётся версионируемым.

**Движок чист.** Решение «БД» — на границе данных, не в движке:
- `engine/chargen` принимает in-memory `RulesData` (дата-классы), считает без I/O.
- **Репозиторий правил** `load_rules_data(session) -> RulesData` собирает `RulesData` из БД.
- Юнит-тесты движка используют in-memory фикстуры — без БД и сети.

## 3. Структура модуля

`backend/app/engine/chargen/` (чистые функции):
- `rules_data.py` — дата-классы `Ancestry`, `Heritage`, `Background`, `ClassDef`, `Feat`, `SpellcastingDef`, контейнер `RulesData`.
- `build_state.py` — `BuildState`: выборы игрока (иммутабельно).
- `abilities.py` — повышения/изъяны характеристик.
- `proficiency.py` — ранги, бонус владения, статическая таблица навыков.
- `derived.py` — производные статы (HP, спасы, восприятие, СЛ, AC, атаки).
- `validation.py` — проверки выборов и пререквизитов.
- `sheet.py` — `CharacterSheet` + экспорт в JSON.
- `compile.py` — `compile_character(rules, build) -> CompileResult` (оркестрация, сбор ошибок).

Загрузка из БД (`load_rules_data`) и сидер — вне пакета `engine` (в `db`/`tools`), чтобы движок оставался чистым.

## 4. Модель данных

Две модели, разделены намеренно:
- **`BuildState`** (выборы): `ancestry_slug`, `heritage_slug`, `background_slug`, `class_slug`, `ability_boost_selections` (назначения свободных повышений по наборам), `voluntary_flaws` (опц.), `skill_selections` (доп. обучаемые навыки), `feat_selections` (родовой/классовый фит), позже — `spell_selections`.
- **`CharacterSheet`** (результат): итоговые характеристики + модификаторы, HP, размер, скорость, трейты, чувства, языки, владения (ранг + значение) по спасам/восприятию/AC/атакам/классовой СЛ/навыкам, выбранные фиты, опц. секция заклинательства.
- `compile_character(rules, build) -> (sheet | None, list[ValidationError])` — чистая функция.

## 5. Характеристики (`abilities.py`)

Все стартуют с 10; наборы применяются в порядке создания:
1. **Предыстория** — 2 повышения (обычно «одно из двух названных + свободное»), в разные характеристики.
2. **Род** — фиксированные повышения + 1 свободное + фиксированные изъяны.
3. **Класс** — 1 повышение в ключевую характеристику (или свободное).
4. **Четыре свободных** — в 4 разные характеристики.

Правила: повышение = **+2 при значении ≤17, +1 при ≥18** (апекс-кэп, проверяется в момент применения); изъян = −2; внутри одного набора повышения — в разные характеристики. Модификатор = `(score − 10) // 2`. **Добровольные изъяны** — опционально, по умолчанию выключены (точные правила доспецифицируются позже; v1 не усложняем).

## 6. Владения (`proficiency.py`)

Ранги: untrained / trained / expert / master / legendary. Бонус: `0 если untrained, иначе level + rank_value`, где T=2, E=4, M=6, L=8. Итог стата = `ability_mod + prof_bonus(rank, level)` (предметные бонусы — позже).

Статическая константа — **16 навыков PF2e** и их ключевые характеристики: Acrobatics(Dex), Arcana(Int), Athletics(Str), Crafting(Int), Deception(Cha), Diplomacy(Cha), Intimidation(Cha), Medicine(Wis), Nature(Wis), Occultism(Int), Performance(Cha), Religion(Wis), Society(Int), Stealth(Dex), Survival(Wis), Thievery(Dex); + Lore(Int, особый, может быть несколько).

## 7. Производные статы (`derived.py`)

Уровень 1:
- HP = `ancestry.hp + class.hp_per_level + Con_mod`
- Восприятие = `Wis_mod + prof(perception_rank)`
- Спасы Fort/Ref/Will = `{Con,Dex,Wis}_mod + prof(save_rank)`
- Классовая СЛ = `10 + key_ability_mod + prof(class_dc_rank)`
- AC = `10 + Dex_mod + prof(unarmored_rank)` — **v1 без доспеха**
- Атаки = `ability_mod + prof(weapon_rank)` (конкретное оружие — позже)
- Скорость = `ancestry.speed`; размер/трейты/чувства/языки — из рода.

**Схема `ClassDef.initial_proficiencies`** (курируется по Fighter сейчас; остальные 21 — поток 2):
`perception` (ранг), `saves{fortitude, reflex, will}`, `defenses{unarmored, light, medium, heavy}`, `attacks{unarmed, simple, martial, advanced}`, `class_dc`, `skills{trained_count_base: int, fixed: [слаги]}`. Число обучаемых навыков = `trained_count_base + Int_mod`.

## 8. Валидация (`validation.py`)

Движок возвращает структурированные ошибки (не бросает исключения):
- Легальность выборов: слаги существуют в `RulesData`; наследие принадлежит выбранной родословне.
- Повышения: количество назначенных свободных совпадает с положенным; внутри набора — разные характеристики; кэп соблюдён.
- Навыки: выбрано ровно `trained_count_base + Int_mod`; без дублей с уже обученными (класс/предыстория).
- Фиты: выбранный фит есть в списке 1 ур. соответствующей сущности; **пререквизиты** проверяются базово (по характеристике/навыку); сложные — помечаются непроверяемыми, не блокируют.
- Неполнота данных: если у класса нет `initial_proficiencies` — ошибка «данные класса не заполнены», лист не компилируется (без молчаливого неверного расчёта).

## 9. Заклинательство — точка расширения

`ClassDef.spellcasting: SpellcastingDef | None` (мартиалы — `None`). `SpellcastingDef` (заполняется потоком 3): традиция (arcane/divine/occult/primal), тип (подготовка/спонтанные), ключевая характеристика, слоты/заговоры 1 ур. `BuildState` получит опц. `spell_selections`; `CharacterSheet` — опц. секцию заклинательства. В v1 ядра поле присутствует, логика выбора заклинаний — заглушка (поток 3).

## 10. Экспорт (`sheet.py`)

`CharacterSheet` сериализуется в **JSON** (собственный формат: характеристики, модификаторы, производные, владения с рангами, фиты, чувства/языки). Печатный лист и Pathbuilder-совместимость — позже.

## 11. Тестирование

TDD, детерминированно, на in-memory фикстурах (без БД и сети):
- Повышения/изъяны (правило ≥18, «разные в наборе», модификатор), бонус владения по рангам, HP/спасы/восприятие/классовая СЛ/AC, выбор навыков, пререквизиты фитов.
- Полный сценарий: собрать **Fighter-дварфа** из seed-`RulesData` → проверить итоговый лист поле-в-поле.
- Репозиторий/сидер (поток 4) тестируются отдельно (интеграционно, с тестовой БД) — вне юнит-тестов движка.

## 12. Порядок реализации

1. **Ядро движка** (этот спек) на seed-фикстурах (Fighter + дварф + одна предыстория) — самостоятельный, тестируемый.
2. **Схема `rules` + сидер** (инфра-поток) — модели, миграции, загрузка JSON → БД, `load_rules_data`.
3. **Данные классов** (поток 2) — извлечение начальных владений/особенностей всех 21 класса.
4. **Заклинания** (поток 3) — данные + логика выбора.
5. Далее — API/персистентность, UI-конструктор (последующие вехи-спеки).
